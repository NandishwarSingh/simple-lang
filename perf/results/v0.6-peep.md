## Analysis (hand-written)

Two peepholes from the Go investigation, plus a methodology fix.

### `cbz`/`cbnz` fusion (arm64/emit.c)

A block ending in `cmp Rn, #0` followed by an eq/ne branch is exactly
ARM64's `cbz`/`cbnz` — two instructions collapsed into one. Detected at
emission time, where both halves are visible.

**primes: 0.19 → 0.17 s, now 1.00x C — Simple beats clang on this
benchmark** (C 0.18 s over 11 runs). The inner loop went from 9
instructions to 8, matching Go's count.

A bug worth recording: the first version inverted the condition, because
`ctoa[c][1]` in QBE's arm64 emitter holds the *negated* string, not the
positive one. The symptom was `make test` hanging — an infinite loop —
which is exactly the failure mode a correctness gate should catch, and did.

### Stack-slot address CSE (arm64/isel.c)

Stock QBE emitted a fresh `add xN, x29, #off` at *every* use of a stack
slot, so an inner loop touching `a[j]` and `a[j+1]` recomputed the array's
base address twice per iteration. Now one address per slot per block,
flushed at the top of the block where it dominates every use.

sortint's inner loop: **12 → 10 instructions** (both fixes combined).
Wall time unchanged at 0.05 s — the loop is bound by the remaining
address arithmetic, which is what indexed addressing (Tier 2 #6) removes.

### Methodology fix: RUNS 5 → 9

The first lab run after these changes showed strbuild at 1.86x and
chanping at 1.07x, which looked like a regression. Re-measuring with 11
runs gave 1.33x and 1.07x — the "regression" was noise. At 5 runs the
sub-100ms benchmarks swing 20–40% between labs, enough to draw a false
conclusion. The harness now takes 9 runs.

**This is a caution about every earlier number in the sub-100ms range**
(strbuild, chanping, sortint, matmul, sieve, bitops): differences under
~20% there were never significant.

### The finding that matters: matmul flips

matmul is **2.50x C on arm64** but **0.71x C on x86_64** — the same
source, the same MIR, the same optimizer. The difference is entirely in
the backend:

```
Simple's x86_64 matmul: 11 indexed-addressing instructions
Simple's arm64  matmul:  0
```

QBE selects addressing modes for amd64 and not at all for arm64. That is
direct, measured proof that **indexed addressing is the top remaining
arm64 item** — not an estimate.

### Attempt and revert

ARM64 register-offset addressing was implemented and **reverted the same
session**. Folding `base + idx*sz` into a `Mem` by nop-ing the defining
`add`/`mul` is unsound: a defining instruction can live in a *different
block*, and the operands it referenced are not necessarily live at the
use. The result segfaulted (`exit 139`) and the golden tests caught it
(11/11 → 9/11) before it went anywhere near a commit.

amd64 avoids the problem with a per-block address numbering
(`anumber`/`amatch`) that *reconstructs* the address instead of deleting
instructions. Porting that to arm64 is a dedicated task, not a peephole —
now noted in `arm64/isel.c` and in the optimization plan.

Repository restored: 11/11 golden, 4/4 safety, differential clean on all
18 programs, x86_64 11/11.
