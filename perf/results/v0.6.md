## Optimization round (post-v0.6)

Worked the hit list top-down, measuring each step.

### 1. Range analysis → bounds-check elimination (the big one)

The v0.6 eliminator only understood `for`-range variables. Real code
indexes with *mutable cursors* (`while (j >= 0 && a[j] > v)`), so
sortint kept 6 of 8 checks. Replaced it with a proper integer range
analysis, built on three language guarantees:

- **No aliasing in safe code** — a function call cannot modify a local,
  so ranges survive across calls with no interprocedural analysis.
- **Guards are proofs** — a condition holds inside what it guards,
  including across `&&` short-circuits (the right side of `a && b` may
  assume `a`).
- **Monotonic loop cursors** — a variable a loop body only ever
  decrements keeps its *upper* bound across the whole loop (and vice
  versa), so `j` stays provably `<= 798` while the guard supplies
  `j >= 0`. No fixpoint iteration needed.

Escape hatch for safety: any variable whose address is taken with `&`
anywhere in the function is excluded, because an `unsafe` write through
that pointer is invisible to the analysis.

| benchmark | checks before → after | time | vs C |
|-----------|----------------------:|-----:|-----:|
| sortint | 8 → 3 | 0.12 → **0.05** | 4.00x → **1.67x** |
| sieve   | 8 → 0 | 0.05 → **0.04** | 1.25x → **1.00x** |
| matmul  | 8 → 0 | unchanged        | 2.50x |
| nbody   | 8 → 0 | —                | — |

**sieve now ties C, Rust and Go exactly.** sortint is 2.4x faster.

Four adversarial programs are now permanent tests
(`tests/safety/`, run by `make test`): an index from an unknown function
parameter, a loop that overruns, a `for` range wider than the array, and
— the subtle one — a variable mutated through an `unsafe` pointer. All
four still trap correctly at runtime.

### 2. Return-copy elision

`return <aggregate literal>` now builds directly into the caller's return
slot (safe unconditionally: that slot is always a fresh temporary, so it
cannot alias what the literal reads). nbody 0.23 → 0.22 s — real but
small, because the copies were not the bottleneck.

### 3. nbody's real gap: vectorization — not closable today

Measured rather than assumed: clang emits **80 SIMD register references
and 23 NEON instructions** for nbody; Simple emits **zero**. The 5.5x is
auto-vectorization, which is an LLVM-scale project, not an optimization
pass. Recorded as a known, bounded gap. The same explanation covers
matmul (2.50x).
