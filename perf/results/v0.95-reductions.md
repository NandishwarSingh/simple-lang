# v0.95 — vectorized reductions

Date: 2026-07-22 · Machine: Apple M5 / arm64 macOS · 7-run median wall time.

Milestone F of the vectorization engine: single-statement f64 reductions
(`s = s + a[i]`, `d = d + a[i]*b[i]`, `p = p * (1+a[i])`) now accumulate
into a 2-lane vector with a horizontal combine, instead of a scalar
left-to-right fold. Reassociation across lanes is licensed by the v0.95
determinism decision (cross-architecture identity is the guarantee, not a
bit-match with today's scalar order).

## Dot product — `sum(a[i]*b[i])` over 4096 elements × 200,000 reps

| build | time | result | notes |
|-------|------|--------|-------|
| **Simple, vectorized + 4-way unrolled** | **0.10s** | 200793929 | deterministic, no flags |
| Simple, single-accumulator (before unroll) | 0.25s | 200793929 | latency-bound |
| Simple, `--no-opt` (scalar) | 0.47s | 200793929 | |
| C `-O2` | 0.38s | 200793929 | reduction stays scalar |
| C `-O3 -march=native` | 0.39s | 200793929 | still scalar — no reassoc |
| C `-O3 native -ffast-math -ffp-contract=off` | 0.06s | 200793929 | vectorized + unrolled, no FMA |
| C `-O3 native -ffast-math` | 0.08s | 200793929 | + FMA (no faster here) |

**Simple beats C by ~4× at its safe/default settings** (0.10s vs
0.38s / 0.39s). A C compiler will not reassociate a floating-point
reduction without `-ffast-math`, so it leaves the dot-product loop
scalar. Simple's determinism model *defines* the reassociation as legal,
so the loop both vectorizes and unrolls with independent accumulators.

Two things this measurement settled about the remaining gap to C's
`-ffast-math` build:

- **FMA is not the lever.** With FMA off (`-ffp-contract=off`) C runs
  0.06s; with FMA on, 0.08s — neutral to slightly *slower*. QBE has no
  FMA op anyway, and it is a 3-input operation that does not fit QBE's
  2-argument instruction, so it was correctly skipped.
- **The lever is unrolling with multiple accumulators**, which breaks
  the latency-bound dependency chain. That took the single 2-lane
  accumulator (0.25s) to four independent accumulators (0.10s).

The residual 0.10s → 0.06s is memory-resident accumulators (a
store/load round-trip per iteration). Register-resident accumulators via
phi nodes were implemented and reverted: under the `Kd` vector shortcut
they are unsafe, because when QBE cannot coalesce a phi it resolves it
with a 64-bit copy that drops a vector's high lane — correct only by
luck when coalescing happens to succeed (it did for this dot product at
0.08s, but the product reduction and all of amd64 came out wrong).
Closing that gap needs the real 128-bit vector class, deferred as
high-risk. 256-bit AVX would help only on amd64 — arm64 NEON is
128-bit — so it is irrelevant to the M-series host.

Tuning: UNROLL 2 → 0.13s, **4 → 0.10s**, 6 → 0.10s, 8 → 0.13s.

## Determinism

Every emitted reduction is **bit-identical across arm64 (NEON), amd64
(AVX via Rosetta), and the scalar path**. A raw 15-significant-digit
probe (`sum(a[i]*a[i] - a[i])`, 777 elements) returns `256819817988306`
on all three. The vectorstorm checksum (whose final sum is now a
vectorized reduction) is unchanged at `811875086734` on both targets.
