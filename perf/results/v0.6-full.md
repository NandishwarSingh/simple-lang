## Reading the table

**Simple places 6th of 7 on raw speed (1.454x C) but 1st–3rd on every
other axis** — memory ties C exactly, binaries are second-smallest, and
compile time is tied-fastest.

Head-to-head, Simple ties or beats: **Swift on 8/11**, **Go on 6/11**,
C and C++ on 3/11, Rust and Zig on 2/11.

Specific results worth noting:

- **bitops splits the field 4–3.** Simple, C, C++ and Zig finish in
  0.03 s; Rust, Swift and Go all take ~0.40 s — 13x slower. Simple's
  loop-idiom recogniser puts it in the fast group, ahead of Rust and Go
  on pure bit manipulation.
- **chanping: Simple 0.33 s vs C 0.34 s.** Our emitted channel runtime
  edges out hand-written C pthreads code. Go's 0.02 s remains in a class
  of its own — goroutine parking versus kernel thread wakeups.
- **sieve: a seven-way tie** (Zig marginally ahead at 0.03 s).
- **Swift is the interesting comparison** — it is the other ARC language,
  and Simple beats it on 8 of 11 benchmarks including nbody (0.23 vs
  0.35) and matmul (0.05 vs 0.25). Swift's array bounds checks and
  copy-on-write checks cost it heavily in tight loops.
- **Zig edges out C overall** (0.983x) — unsurprising for a language
  designed as a better C with the same LLVM backend.

### Fairness notes

- Every implementation is the same algorithm, written naturally for each
  language, with `-O2`-class flags. All 7 produce byte-identical output
  on all 11 benchmarks.
- The Zig and Swift programs were written for this comparison and may not
  be what an expert in those languages would write. One error was caught
  and fixed: Zig's strbuild originally used the debug-oriented
  `GeneralPurposeAllocator` and took 11.10 s; with `c_allocator` it takes
  0.12 s. Others may remain.
- Go wins primes and sortint outright; those deserve investigation rather
  than dismissal.

## Investigation: why Go appeared to beat C

Two anomalies from the table were followed up. One was noise; one is real.

### sortint — NOT REAL, measurement noise

At 11 runs each instead of 5, C, Go and Rust all land on **0.02 s**. The
earlier "Go 0.02 vs C 0.03" was quantisation at the bottom of the timer's
useful range. No investigation needed; the benchmark simply needs to be
scaled up before its numbers mean anything.

**Simple's 0.05 s here is real**, and the cause is visible in the inner
shift loop (12 instructions):

```
add x6, x0, #1          ; j+1
lsl x8, x6, #3          ; (j+1)*8
cmp w9, #0
beq L17
add x6, x29, #16        ; array base
add x6, x6, x7          ; + j*8
ldr x6, [x6]            ; a[j]
add x7, x29, #16        ; array base AGAIN
add x7, x7, x8          ; + (j+1)*8
str x6, [x7]            ; a[j+1]
sub x0, x0, #1
b   L12
```

Two clear defects, both already on the plan:

1. **No indexed addressing.** `ldr x6, [base, x0, lsl #3]` would replace
   `lsl` + `add` + `add` + `ldr`. ARM64 has the addressing mode; QBE's
   ARM64 backend never selects it (only amd64 does). — *Tier 2, item 6.*
2. **The frame-relative base address is recomputed for every access**
   (`add x6, x29, #16` appears twice in one loop body). Straightforward
   CSE of an address that is loop-invariant by construction.

Fixing both should take that loop from 12 instructions to ~7.

### primes — REAL, and it is Go beating clang

Stable across 15 runs: **Go 0.15 s, C 0.18 s, Simple 0.19 s** (variance
±0.01). Both compile the inner loop to 8 instructions containing one
64-bit division:

| | clang | Go |
|---|---|---|
| loop control | keeps `d+1`, recomputes `d` each iteration (`sub`) | keeps `d` directly |
| divide | `udiv` (proved non-negative) | `sdiv` + a `CBZ` divide-by-zero guard |
| order | divide, then bound check | bound check, then divide |

Honest conclusion: the two are instruction-count equal, and this loop is
dominated by a ~10-cycle divide, so the 17% almost certainly comes from
scheduling around that divide rather than from any single visible defect.
Without hardware performance counters this cannot be attributed further —
recorded as an open question rather than a solved one. **The useful part
of the finding is that C is not automatically the ceiling.**

### Simple's own primes loop — 9 instructions vs Go's 8

```
L5: mul x3, x1, x1      ; d*d
    cmp x3, x0
    bgt L9              ; exit
    sdiv x17, x0, x1
    msub x3, x17, x1, x0
    cmp x3, #0          ; <- `cbz x3, L8` would be one instruction
    beq L8
    add x1, x1, #1
    b   L5              ; <- unconditional back-branch: no loop rotation
```

Two fixable items:

3. **`cmp reg, #0` + `b.eq/b.ne` → `cbz`/`cbnz`.** A pure peephole; ARM64
   has had these since day one. Helps every zero-test branch in the
   language. — *new item, small effort.*
4. **Bottom-tested loops.** The unconditional `b L5` disappears if the
   test moves to the bottom. Already on the plan with the warning that it
   must run *after* if-conversion in the backend, not before. —
   *Tier 3, item 12.*
